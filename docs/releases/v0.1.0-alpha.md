# v0.1.0-alpha

`memory-runtime` is now ready for a public GitHub-first release.

## What this release includes

- local SQLite hot memory
- Memory Palace cold-memory adapter
- Codex, Claude, and Gemini wrappers
- automatic shell integration for `zsh` and `bash`
- backend-only Docker deployment path for cold memory
- fail-open bootstrap behavior
- public-facing docs for install, configuration, troubleshooting, and release flow

## Why this exists

Most CLI agent sessions waste tokens by replaying too much history or by
injecting context too aggressively.

`memory-runtime` is built around a stricter rule:

- keep the raw user request intact
- inject only compact, stable background context
- reduce memory injection when risk is high
- never block the CLI when memory is unavailable

## Recommended first run

```bash
git clone https://github.com/Tom-Wang898/memory-runtime.git
cd memory-runtime
npm install
./scripts/install-shell-integration.sh --shell zsh
source ~/.zshrc
hmctl bootstrap --cwd "$(pwd)" --mode warm --query "runtime smoke test" --json
```

If you want cold memory with the least setup friction:

```bash
./scripts/install-memory-palace-docker.sh
source ~/.memory-runtime/env.sh
```

## Current scope

This release is `0.x` on purpose.

It is stable enough for GitHub users who are comfortable cloning a repo and
running local scripts, but it is not positioned yet as an npm-published product.

## Known boundaries

- current shells still need one reload after shell integration is installed
- already-running CLI processes cannot be retrofitted
- Docker cold memory is operator-managed; wrappers do not start Docker automatically
- the full Memory Palace dashboard stack remains outside this repo

## Feedback focus

The most useful feedback for this release:

- wrapper behavior across real Codex / Claude / Gemini workflows
- cold-memory relevance quality
- installation friction on clean machines
- token savings vs. context drift tradeoffs
